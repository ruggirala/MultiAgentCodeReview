Why this exists

Code review is the bottleneck no team escapes.

~26h

Reviews are slow

SmartBear's State of Code Review found PR turnaround averages over a day. Critical fixes wait while reviewers context-switch.

~70%

Reviews miss things

Studies of human code review show roughly 30–40% of defects slip past two reviewers — and security issues slip more often than functional bugs.

N×

Reviews don't scale

Senior engineers are scarce. Every junior PR competes for the same handful of approvers — review quality degrades under load.

A focused multi-agent system can do the first pass in under a minute, ground every finding in CWE/OWASP, and free reviewers to focus on architecture.

Per-PR pipeline flow

From PR open to posted comment, one shared state object visits each agent in turn. Median timings shown are from real load-test telemetry.

orchestrate ~0.0s
security 3.5s

human_review* conditional
patch 21.0s

Shared ReviewState

Pydantic v2 model threaded through every node. Every agent reads source_code, appends Findings, and (downstream) populates patch and tests. No string parsing between agents.

Conditional routing

The triage node sets needs_human_review = True when any Critical/High security finding exists. LangGraph's conditional edge then routes through human_review before patching — a real interrupt point in production.

* human_review fired on 25 / 31 pipeline runs in the load test — most PRs had Critical security findings.

Telemetry & dashboard

Every run emits structured events — a Streamlit app tails them in real time.

Four event types

pr_review
one per PR · counts, severities, duration
agent
one per LangGraph node · timing, errors
llm_call
one per call_llm() · backend, tokens, retries
poll_cycle
one per watcher poll · throughput, errors

Append-only line-delimited JSON in runs/events.jsonl. No DB, crash-safe, trivially tailable. contextvars.ContextVar attributes every LLM call back to the agent that made it.

Dashboard surfaces

1

Overview KPIs with sparklines

PRs reviewed · findings · throughput · agent error rate · avg review duration. Dashboard's purple-blue gradient cards.

2

Agent-interaction Sankey

Aggregated edge counts across runs — exactly which paths the LangGraph state machine took.

3

Per-agent duration box-plot

Distribution of latency by agent. Catches outliers like a tree-sitter parse blow-up.

4

LLM telemetry

Calls per backend/model, total tokens, retry rate, p95 latency.

Live demo flow

Three terminals + your browser. The PR is the only thing the user does — the rest is automatic.

1

Start the watcher

SKIP_CI_GATE=1 python watch_prs.py rahulilla/airflow — pre-seeds existing open PRs, polls every 20s.

2

Start the dashboard

streamlit run dashboard/app.py — opens at localhost:8501, auto-refreshes every 10s.

3

Open a PR on the fork

Any .py change on rahulilla/airflow. Within one poll cycle, the watcher detects it and runs the multi-agent pipeline.

4

Watch the artifacts appear

Comment posts on the PR with severity table + every finding · Markdown report + ZIP saved locally · sibling branch with the agent's fixed code · follow-up PR opens automatically.

5

Refresh the dashboard

New event lands in events.jsonl → KPIs tick up → Sankey edge thickens → LLM tab shows tokens spent.
